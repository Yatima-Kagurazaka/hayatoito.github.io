appendix

firstChild, lastChild は Node クラスではなく、Node クラスを継承した ContainerNode クラスに定義されています。 例えば、子供を持つことができない、末端の Node である Text Node (テキストノード) などは、このようなポインタを持つ必要がありません。 以下は、この違いを無視して、すべて Node として扱います。

getElementById(xx) vs querySelector('#xx')

querySelector('#xx') は getElementById(xx) と同じ動作になります。querySelector の場合は、任意のCSSセレクタがパラメータとして 渡りうるので、一見、getElementById(xx) より遅いと思われるかもしれません。 実際は、querySelector('#xx') は getElementById(xx) とほぼ同じ速度で動作します。 このように非常によくあるケースでは、ブラウザは内部で shortcut を用意して、通常のCSS Selector マッチングの処理ではなく、特別なショートカットパスに移ることはよくあります。 querySelector は渡された CSS セレクタが '#id' の形式であることが判明しだい、...。 この手の「ショートカット」はありとあらゆるところにいれています。

オーバーフローを気にしない。

例: 手抜きアルゴリズム / Cascading overflow: id class 少しの意図的な悪いサイトのために、善良なよいサイトがパフォーマンスのペナルティを受ける必要はない。 ある特定のベンチマークのスコアを伸ばすために、普通な善良なサイトがペナルティを受けるなら、それには反対します。

Shadow DOM is Monad。

Shadow DOM が モナドであるという恐ろしい証明に私は成功しましたが、その証明を書くにはここは余白が狭すぎます。

下のレイヤーの人が「がんばる」ことによって、上のレイヤーの人の悩みが一挙に解決したりすることは非常によくあります。 たとえば、jQuery は多くの Web 開発者から、DOM 操作の苦しみを和らげることに成功しました。 次に起きたのは、Web プラットフォームそのもにに、Node.append や Node.before が実装されることになり、jQuery そのものを使用する必要性が大幅に減りました。 これは、「Web 開発者 => フレームワーク => ブラウザ」とどんどん下のレイヤーで問題を解決することで多くの人が幸せになれるよい例です。 ここで大事なのは意外に思うでしょうが「コミュニケーション」です。あまり仲間内だけで悩んでばかりいないで、たまには「外の世界」の人に悩みを相談すると、一挙に解決してくれるかもしれません。 ただし、Web プラットフォームのレイヤーで問題を解決することが常にベストとは限りません。これについては後に述べます。
過去にいろいろなプラットフォームが、Webに挑んでいっては、敗れ去っていきました。Flash, Silverlight等。 中には、Web より過ぐれた技術もあったかもしれませんが、人々はWebを選び、これらのことごとく破れていった事実は、謙虚に受け止めてもよいかもしれません。

poem

Web プラットフォームは、他のプラットフォームと比べて「変化が伝わるスピードが早い」です。すなわち、我々がちょっとミスをしちゃうと、あっという間に、ユーザーの Web サイトが壊れます（そして怒られます）。 他のプラットフォームであれば、例えば、SDK の API を変更したとしても、古いアプリがすぐに動かなくなるわけではありません。 一方、ブラウザに対する変更は、あっという間に、10億人以上のユーザーに配布され、その結果、「Webサイトがなにもしてないのに壊れた！」が起きてしまいます。 実際に、我々はかなりの割合で事故として「Webを壊して」しまいます。そしてすぐに直します。:)

「Web を進化させる」というのは、ユーザーが10億人以上いるそびえたつジェンガの山を、崩れないようにピースをとっていき、ぴっかぴかのプラットフォームに生まれ変わられること。しかも、これを10億以上のユーザーを前に気づかれないようにこっそり行うこと。
